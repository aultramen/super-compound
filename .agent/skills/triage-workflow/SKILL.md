# Triage Workflow

## Purpose

Move raw requests and existing issue files through a small state machine until each item is either agent-ready, human-owned, waiting for information, or intentionally rejected.

Announce: "I'm using the triage-workflow skill to classify and prepare issues."

Use this behind `/sc-plan` and `/sc-status`; do not add a standalone triage command.

## State Model

Category roles:

- `bug`: something is broken
- `enhancement`: new or improved behavior

State roles:

- `needs-triage`: maintainer needs to evaluate
- `needs-info`: waiting on reporter or owner
- `ready-for-agent`: fully specified and agent-ready
- `ready-for-human`: requires human judgment, access, or approval
- `blocked`: blocked by unresolved `OPEN-*`, missing FSD authority, or unavailable required access
- `wontfix`: will not be actioned

Each issue should have one category and one state. If states conflict, stop and ask before changing the issue.

## Process

### 1. Gather Context

Read the issue body, comments, linked FSD/PRD/BRD, related `.scratch/` board, relevant code, tests, `CONTEXT.md`, and linked accepted ADRs.

Check two things before recommending work:

- Redundancy: is the requested behavior already implemented?
- Prior rejection: is the same enhancement already recorded under `.out-of-scope/` or project docs?

### 2. Recommend A State

Report the category, state, and short reasoning.

For bugs, reproduce or identify the tightest available verification signal before marking `ready-for-agent`.

For enhancements, ensure the behavior, non-goals, acceptance criteria, and verification are concrete enough for an agent.

### 3. Clarify When Needed

If the issue is underspecified, ask one focused question or move it to `needs-info` with:

```markdown
## Triage Notes

**Established:**
- <known fact>

**Needed:**
- <specific question>
```

Use `domain-modeling` when terminology is fuzzy or disputed.

### 4. Produce Agent Briefs

For `ready-for-agent`, add or update the issue with:

- What to build or fix
- Acceptance criteria
- Verification command or manual proof
- `Blocked by` relationships
- Relevant constraints from FSDs, PRDs, BRDs, linked accepted ADRs, or domain docs

Do not triage issues generated by `issue-workflow` unless they changed, became stale, or failed verification; those issues are already intended to be agent-ready.

### 5. Handle Rejections

For `wontfix`:

- Explain the reason concisely.
- If it is an already implemented request, point to the behavior and do not write out-of-scope memory.
- If it is a rejected enhancement, record the durable reason under `.out-of-scope/<concept>.md` when future duplicate requests are likely.

## Local Markdown Updates

When updating `.scratch/<feature>/issues/*.md`, preserve existing comments and append triage notes under `## Comments`. Do not erase user-provided history.

- `systematic-debugging` verifies bugs
